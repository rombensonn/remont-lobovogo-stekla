# Glass Atelier Master Specification

Мастер-спека описывает визуальную систему для сайта `[НАЗВАНИЕ]`: ремонт сколов и трещин, тонировка, растонировка и бронь лобового стекла. Это документационный контракт для параллельной работы над UI; он не должен добавлять новые бизнес-факты.

## Non-Negotiables

- Редактировать визуальную систему через `DESIGN.md`, `design-tokens.json` и этот файл.
- Не выдумывать `[АДРЕС]`, `[ТЕЛЕФОН]`, `[EMAIL]`, `[WHATSAPP]`, `[TELEGRAM]`, `[ГОРОД]`, `[РАЙОН]`, `[ИП/ООО]`, `[ИНН]`, `[ОГРН/ОГРНИП]`, гарантии, бренды пленок, сертификаты и реальные кейсы.
- Использовать радиус `8px` как базовый для кнопок, полей, техкарт и модальных поверхностей.
- Проектировать mobile-first: все ключевые действия должны быть доступны на узком экране без горизонтального скролла.
- Соблюдать WCAG AA для текста, видимые focus-состояния и touch target не меньше `44px`.

## Creative North Star

Сайт должен ощущаться как **glass atelier**: чистый рабочий пост по автостеклам, где мастер смотрит на повреждение под лампой, отмечает трещину измерительными линиями и аккуратно объясняет следующий шаг. Это не generic автосервис и не карточный SaaS-лендинг.

Визуальные ассоциации:

- графитовая приборная панель;
- фарфоровая чистая поверхность стола;
- циановый край стекла и оптический блик;
- сдержанный янтарный маркер цены или предупреждения;
- техническая карта с параметрами, линиями и короткими заметками.

## Color System

База строится на четырех группах из `design-tokens.json`:

- `graphite`: навигация, инверсные секции, основной текст, тонкие технические акценты.
- `porcelain`: общий фон, формы, спокойные рабочие секции.
- `cyan`: primary CTA, focus ring, выбранные состояния, линия стекольной оптики.
- `amber`: цены, ограничения, предупреждения, но без распродажного крика.

Правило баланса: porcelain и graphite создают 80-90% интерфейса, cyan задает действие, amber используется точечно.

## Typography

Шрифт по умолчанию: системный sans из токенов. Моноширинный стиль допустим для коротких технических меток: размер скола, тип услуги, статус заявки, номер шага. Не использовать отрицательный letter-spacing. Заголовки могут быть крупными, но внутри техкарт и форм типографика должна быть компактной и сканируемой.

Рекомендуемая иерархия:

- Hero/title: 36-48px desktop, 30-36px mobile.
- Section heading: 28-36px desktop, 24-30px mobile.
- Body: 16-18px.
- Labels/captions: 12-14px, uppercase только для коротких технических меток.

## Layout System

Сетка:

- mobile: 4 колонки;
- tablet: 8 колонок;
- desktop: 12 колонок;
- max content width: `1180px`.

Секции должны быть full-width полосами или простыми constrained layouts. Не вкладывать карточку в карточку. Hero должен сразу показывать предмет сайта: автостекло, ремонт, тонировка или техническую схему стекла, а не абстрактный маркетинговый экран.

## Surface Language

Допустимые поверхности:

- porcelain page background;
- graphite inverse band;
- glass surface с умеренной прозрачностью и blur;
- hairline separators;
- technical sheet / service sheet с параметрами.

Недопустимые паттерны:

- одинаковые floating cards ради декоративной сетки;
- массивные gradient blobs;
- purple/blue SaaS-градиенты;
- overly rounded pill-интерфейс;
- декоративные бейджи, которые выглядят как неподтвержденные гарантии.

## Technical Graphics

Измерительные линии являются частью языка:

- `1px` линии с низким контрастом;
- короткие ticks по краям блоков;
- координатные подписи только когда они помогают сканировать;
- диагональные/радиальные линии допустимы для схемы скола или трещины;
- графика не должна мешать чтению, клику или фокусу.

Примеры текстовых меток без выдуманных фактов:

- `ремонт скола`;
- `трещина`;
- `тонировка`;
- `растонировка`;
- `бронь лобового стекла`;
- `ориентир по стоимости`;
- `заявка`.

## Component Guidance

### Buttons

Primary: cyan background, graphite text or very dark text при достаточном контрасте. Secondary: transparent/porcelain with graphite border. Destructive or warning states use amber/danger only when пользовательское действие действительно рискованное.

### Forms

Форма заявки должна быть максимально ясной: имя, телефон, услуга, фото/комментарий если уже предусмотрено, два обязательных согласия. Labels всегда видимы. Placeholder не заменяет label. Ошибки показываются рядом с полем и не зависят только от цвета.

### Service Techcards

Техкарта услуги может содержать:

- название услуги;
- короткое объяснение;
- ориентир по цене, если он уже есть в проекте;
- что влияет на расчет;
- CTA на заявку.

Не добавлять гарантию, сроки, бренды материалов или реальные примеры без источника.

### Navigation

Навигация компактная, контрастная, без тяжелого sticky-шума. На mobile действия должны быть доступны большим touch target. Основной CTA может повторяться в нижней части ключевых секций, если это не перегружает страницу.

### Media

Если нет реальных фото, использовать абстрактную, но предметную графику: стеклянная плоскость, линии дефекта, измерительные сетки, силуэт лобового стекла. Не имитировать реальный кейс клиента и не показывать бренды пленок.

## Accessibility Checklist

- Контраст обычного текста соответствует WCAG AA.
- Focus ring виден на светлом и темном фоне.
- Все элементы управления достижимы с клавиатуры.
- Текст не расположен поверх шумного изображения без устойчивой подложки.
- Touch targets не меньше `44px`.
- Motion отключается или упрощается при `prefers-reduced-motion`.
- Форма не полагается только на цвет для ошибок и обязательных полей.

## Implementation Notes

- Токены из `design-tokens.json` являются источником имен и значений.
- Tailwind-классы должны мапиться к этим токенам без расширения палитры случайными цветами.
- Радиус больше `8px` применять только для полностью круглых технических маркеров или аватароподобных элементов, если они реально нужны.
- Не использовать in-app текст, который объясняет дизайн-систему пользователю сайта.
- Любая новая бизнес-информация должна проходить через существующие плейсхолдеры или подтвержденные данные проекта.
